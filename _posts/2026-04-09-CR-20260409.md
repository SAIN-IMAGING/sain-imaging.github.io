## Compte-rendu


### Rusdesk
Dans le cadre du réseau SAIN, Denis Grenier a mis en place un outil **RustDesk** permettant aux utilisateurs de se connecter sur les stations Bruker des membres volontaires.

Son utilisation principale vise à simplifier le développement de séquence ainsi que l'entreaide sur les machines.

Un mode opératoire sur Resana :  <https://resana.numerique.gouv.fr/public/perimetre/consulter/1302885?information=46855085>

Actuellement les centres disposants de Rustdesk sont : 
* Lyon : ok
* Marseille : ok
* Grenoble :
  * problème de proxy -> déplacement des machines sous nouveau sous-réseau
* Bordeaux : ok
  * Ouverture des ports nécessaires

### Hardware seconde main

L'initiative **Mastothèque** semble être une approche qui pourrait convenir car elle peut-être auto-hébergées :

### Metadata

Un groupe de travail EMIM a commencé sur la standardisation des modalités d'imageries

Le SAIN a déjà commencé une réflexion et a généré un document qu'il faudrait compléter :
* Travail initial de définition par le SAIN disponible sur [ici sur RESANA](https://resana.numerique.gouv.fr/public/perimetre/consulter/1302885?information=25936203)

* [LIVRE_BLANC-Utilisation_secondaire_des_donne_s_dimagerie_a__des_fins_de_recherche](https://resana.numerique.gouv.fr/public/perimetre/consulter/1302885?information=49312007)

**Discussion :**
* En IRM préclinique (Bruker) le fichier Method et acqp et visu_pars stocke la plupart des metadatas. Peut-être **TROP** pour envisager un partage ?
* Il n'y a pas de code associé avec le type d'image (foie etc)
* Le type d'antenne semble être une metadata importante
* Les métadatas des fichiers ne sont pas adapté pour l'entraînement IA


Une nouvelle version de brkraw (conversion des données en NIFTI) permet de définir des modules. Il peut-être intéressant de définir une liste permettant de générer un fichier exploitable de métadata  <https://brkraw.github.io/getting-started/bids.html>. **La discussion doit se faire sur le forum Bruker pour obtenir un consensus**
